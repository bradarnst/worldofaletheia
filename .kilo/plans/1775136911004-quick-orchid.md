# Next Work Plan: Gaps, Quick Wins, and Next Feature

## Goal

Recommend the highest-value next steps after recent auth/campaign fixes, balancing fast wins, risk reduction, and meaningful product progress.

## Snapshot (from current plans/docs)

1. Discovery/index foundation is largely implemented, but **Gate G1/G2 closure** is still explicitly open in planning docs.
2. Campaign auth is live, but there is still medium-term structural debt around **Option 3 unified roles**.
3. Calendar ADR is accepted, but calendar/timeline implementation remains only partially complete in code maturity.
4. Repo-level automation and quality signals are still weak:
   - no `.github/workflows/*` present
   - repo-wide `tsc --noEmit` is currently red (multiple unrelated pre-existing failures)
5. Documentation drift risk is acknowledged in status docs; several plans contain superseded priority signals.

## Important Gaps

1. **Operational confidence gap**
   - No CI parity lane means regressions are still caught mostly by manual/operator runs.

2. **Discovery product gap**
   - Infra exists, but formalized taxonomy UX contract and sync failure semantics are not fully closed (G1/G2).

3. **Engineering baseline gap**
   - Type-check baseline is red, which reduces confidence in refactors and review signal quality.

4. **Planning signal gap**
   - Status/plan drift can lead to selecting lower-value work out of order.

## Quick Low-Hanging Fruit (recommended first)

### Q1. Add a minimal CI quality lane (high impact, low effort)

- Add one GitHub Actions workflow for PRs with:
  - `pnpm install --frozen-lockfile`
  - `pnpm test`
  - `pnpm build`
- Keep sync/remote checks manual for now; avoid over-coupling CI to secrets or remote infra.
- **Value:** immediate regression prevention with minimal architectural risk.

### Q2. Create a “known TS failures baseline” and burn it down (high impact, low-medium effort)

- Triage current `tsc --noEmit` errors into buckets:
  1) infra config typing
  2) calendar typing
  3) sitemap/content typing
  4) content-filter typing/tests
- Fix smallest buckets first to restore a trusted static-analysis signal.
- **Value:** faster and safer delivery on every subsequent feature.

### Q3. Doc-priority refresh pass (medium impact, very low effort)

- Update one current status doc to mark:
  - done vs pending items,
  - superseded plans,
  - authoritative next 3 priorities.
- **Value:** avoids execution drift and reduces decision overhead.

## Next Larger Feature (recommended)

## Feature Recommendation: Discovery Closure (G1 + G2 + grouped UX)

### Why this should be next

1. It is already identified as top priority in project status docs.
2. Core index/search plumbing exists; this is mostly contract + UX + hardening closure.
3. It improves value across multiple collections/domains immediately.
4. It de-risks later campaign taxonomy and calendar expansion by stabilizing discovery behavior first.

### Scope

1. Finalize taxonomy UX/data contract (type/subtype/tag filters, grouped view behavior).
2. Implement/complete grouped discovery views per contract.
3. Define and enforce sync failure semantics (fail-fast vs partial success) for authoritative lanes.
4. Add targeted tests for filter/group logic and failure-path expectations.

### Suggested delivery shape

1. **Sprint A (design + contract):** Gate G1/G2 docs + acceptance criteria.
2. **Sprint B (implementation):** grouped view + sync hardening changes.
3. **Sprint C (verification):** focused test pass + runbook updates.

## Alternative Larger Feature (if you want user-facing novelty first)

### Calendar MVP completion

- Good option if immediate product-visible progress is preferred over platform hardening.
- Tradeoff: increases risk if discovery and TS baseline remain unstable.

## Recommended Order

1. Q1 CI lane
2. Q2 TS baseline burn-down (at least first 30-50% of failures)
3. Q3 doc-priority refresh
4. Discovery Closure feature (G1/G2 + grouped UX)
5. Then either:
   - Calendar MVP completion (user-facing), or

1. One-page acceptance checklist for Discovery G1/G2 is agreed.
2. CI lane merged and passing.
3. TS error count reduced to a manageable, tracked set (or fully green).
4. Single “current priorities” status doc is updated as source of truth.
