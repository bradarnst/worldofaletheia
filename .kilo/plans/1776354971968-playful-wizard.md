## Site-Wide Search Architecture Session (Kickoff)

Date started: 2026-04-17

### Final Goal (Locked)

Full site-wide search with **D1 FTS5** is the required end state (not optional).

### Ground Truth Confirmed (Current State)

1. Search foundation exists and is live in code:
   - API: `src/pages/api/search.ts`
   - UI route: `src/pages/search.astro`
   - Header entrypoint: `src/components/GlobalNavigation.astro`
2. Canonical discovery/search backing store is D1 `content_index` (ADR-0011 + ADR-0016).
3. Current query behavior is metadata search (title/summary/slug/type/subtype/tags) using SQL `LIKE`; no body FTS yet.
4. Current `/api/search` behavior is public-scope only by default (`visibilityScope: 'public'`).
5. Current `/search` page is `noindex,nofollow`.

### Architecture Constraints (Must Hold)

1. Astro-native first; no speculative service/adaptor/contract layer.
2. D1 remains canonical lookup/index; R2 remains blob storage only.
3. Access control is deny-by-default for protected content on missing/ambiguous auth context.
4. SEO policy remains selective: index content pages, not search result pages.

### Delivery Strategy Options Toward Mandatory FTS5

#### Option A - Keep S1 and S2 as separate pre-FTS steps

Pros:
1. Lower risk per change-set and easier rollback.
2. API contract/authz correctness can be stabilized before FTS complexity.
3. Produces usable incremental value if FTS migration hits unexpected issues.
4. Easier debugging because failures are isolated by layer (contract, ranking, then FTS engine).

Cons:
1. Slower path to final FTS5 capability.
2. Some relevance logic may be reworked after FTS arrives.
3. More intermediate states to document and maintain.

#### Option B - Skip most of S1/S2 and go directly to FTS5

Pros:
1. Fastest route to the desired final architecture.
2. Avoids investing heavily in interim non-FTS ranking behavior.

Cons:
1. Larger blast radius and higher integration risk.
2. Harder to diagnose whether issues come from contract, authz, ranking, or FTS ingestion.
3. Higher chance of delay if any single deep issue blocks rollout.

#### Option C - Hybrid (Recommended)

1. Keep a **minimal S1** (contract + visibility/authz semantics only).
2. Go directly to **mandatory FTS5 implementation** next.
3. Treat S2 as **post-FTS relevance tuning**, not a precondition.

Why recommended:
1. Preserves the fastest practical route to FTS5.
2. Still de-risks the highest-impact boundary (contract + non-leak authz behavior) before engine changes.
3. Avoids over-investing in interim relevance logic that FTS5 will supersede.

### Updated Phase Model (FTS5 Required)

1. **S1 (minimal, required):** API/search contract + auth-aware visibility policy + stable response envelope.
2. **S3 (required):** D1 FTS5 table/indexing/query path for body-text search, with contract preserved.
3. **S2 (after FTS5):** relevance weighting/snippets/highlighting/tuning and UX quality iteration.
