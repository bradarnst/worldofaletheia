# Contributors MVP Implementation Plan

## Scope

Implement the contributor MVP now that the Obsidian vault has been migrated from `author:` to `authors:`.

This pass should deliver:

1. Repository support for `authors` and structured `contributors` metadata.
2. A new `contributors` content collection and public contributor routes.
3. Footer discovery for `/contributors`.
4. Contributor profile display modes: `standard` and `featured`.
5. Contributor profile search links.
6. Content sync/index support sufficient for the new collection and renamed authorship metadata.

This pass should defer:

1. Image attribution validation.
2. Site-wide single-image lightbox.
3. Gallery/previous-next image state.
4. Full contributor-aware search API semantics beyond a pragmatic MVP search link.

## Current Findings

- `src/content.config.ts` still requires `author: z.string()` in `baseSchema`.
- `src/content/**` is gitignored and currently absent in the repo; local content comes from sync, while production/cloud reads use D1/R2.
- `config/content-sync.config.json` currently has mappings for canon/campaign/meta/assets but no `contributors` mapping.
- `content_index` currently has an `author TEXT` column; no DB column for `authors` or structured contributors exists.
- `scripts/content-sync/cloud-content-metadata.mjs` currently writes `frontmatterRecord.author` into `content_index.author`.
- `scripts/content-sync/validate.mjs` currently requires `author`.
- UI/layout consumers read `data.author` in content headers, cards, utility filters, and campaign session detail code.
- Contributor markdown should be authored in the Obsidian vault, not created as tracked `src/content` files.
- Two untracked one-off files exist from the earlier helper script: `scripts/rename-author-frontmatter.mjs` and `scripts/rename-author-frontmatter.test.mjs`. Since the vault migration is already done and the script was intentionally one-off, remove them during implementation unless the user explicitly wants to keep them.

## Data Model Decisions

### Content Entries

Use:

```yaml
authors:
  - brad
  - barry
contributors:
  - id: example-artist
    roles:
      - artist
      - cartographer
```

Rules:

- `authors` is required for existing base content collections that previously required `author`.
- `authors` is an array of contributor ids.
- `contributors` is optional.
- `contributors[].roles` is an array.
- Do not duplicate authors in `contributors` unless they also performed a separate non-author role that needs explicit attribution.
- Initial contributor roles: `artist`, `editor`, `researcher`, `consultant`, `cartographer`, `photographer`, `other`.

### Contributor Entries

Create a new `contributors` collection with frontmatter like:

```yaml
title: Brad
displayName: Brad
status: publish
avatar: /assets/images/contributors/brad.jpg
bioExcerpt: Short summary for index cards.
socials:
  - label: Website
    url: https://example.com
profileMode: featured
featuredContributions:
  - collection: lore
    slug: example-entry
```

Notes:

- `roleSummary` must not be added.
- `profileMode` values: `standard`, `featured`.
- `featuredContributions` is optional and only used by `featured` profiles.
- Contributor collection entries do not need `authors` or `contributors` fields.

## Implementation Steps

### 1. Clean Up One-Off Rename Script

- Remove `scripts/rename-author-frontmatter.mjs`.
- Remove `scripts/rename-author-frontmatter.test.mjs`.

Rationale: the vault migration is complete, the user found the script too complex for a one-off, and keeping it risks confusing future maintenance.

### 2. Update Shared Schema

Edit `src/content.config.ts`:

- Add shared contributor-role constants near schema definitions or in `src/lib/content-types.ts` if reused elsewhere.
- Replace `author: z.string()` in `baseSchema` with `authors: z.array(z.string()).min(1)`.
- Add optional `contributors` to `baseSchema`:

```ts
contributors: z.array(z.object({
  id: z.string(),
  roles: z.array(z.enum(CONTRIBUTOR_ROLE_TYPES)).min(1),
})).optional().default([])
```

- Add `contributorsSchema` collection schema that does not extend `baseSchema`:
  - `title: z.string()`
  - `displayName: z.string().optional()`
  - `status: z.enum(['draft', 'publish', 'published', 'archive', 'archived'])`
  - `avatar: z.string().optional()`
  - `bioExcerpt: z.string().optional()`
  - `socials: z.array(z.object({ label: z.string(), url: z.string().url() })).optional().default([])`
  - `profileMode: z.enum(['standard', 'featured']).default('standard')`
  - `featuredContributions: z.array(z.object({ collection: z.string(), slug: z.string() })).optional().default([])`
- Add `contributors = defineCollection({ loader: createMarkdownLoader('contributors', '**/*.md', 'src/content/contributors'), schema: contributorsSchema })`.
- Export `contributors` in `collections`.

Implementation note: if `avatar` is intended to support relative Obsidian image paths later, keep it as plain string for now and do not force `z.string().url()`.

### 3. Update Content Type Constants

Edit `src/lib/content-types.ts`:

- Add `CONTRIBUTOR_ROLE_TYPES` for the role enum.
- Do not add contributor collection to `COLLECTION_TYPES` unless contributor entries require a `type` field. The R2 loader only sanitizes `type` when a collection has an entry in `COLLECTION_TYPES`, so omitting contributors is appropriate.

### 4. Update Content Sync Validation

Edit `scripts/content-sync/validate.mjs`:

- Change required key from `author` to `authors`.
- Enhance its lightweight frontmatter parser enough to recognize YAML list values for `authors` similar to current `tags` handling.
- Add basic validation that `authors` is present and non-empty.
- Keep deeper contributor-id validation out of MVP unless straightforward.

Update tests:

- `scripts/content-sync/validate.test.mjs`
- any fixtures containing `author:` should move to `authors:`.

### 5. Update Cloud Metadata and D1 Index Compatibility

MVP approach: keep the existing `content_index.author` column for compatibility, but store a display string derived from `authors`.

Edit `scripts/content-sync/cloud-content-metadata.mjs`:

- Read `frontmatterRecord.authors`.
- Normalize array/string forms defensively.
- Emit `author` in the row as `authors.join(', ')` so existing D1 schema and display code keep working during the MVP.

Reasoning:

- Avoid adding a D1 migration for `authors_json` in this MVP.
- Avoid destabilizing search/discovery/index queries.
- Preserve existing `content_index.author` consumers while source metadata moves to `authors`.

Follow-up after MVP:

- Consider `authors_json` and `contributors_json` columns if contributor-aware search/filtering becomes a concrete requirement.

Update tests:

- `scripts/content-sync/cloud-content-metadata.test.mjs`
- `scripts/content-sync/content-index-writer.test.mjs`
- `scripts/content-sync/content-discovery-writer.test.mjs`

Only update expected values where test fixtures currently use `author`.

### 6. Update Runtime Types and UI Consumers

Update local/index card mapping to use `authors` while preserving the existing display surface as a joined author label.

Files to update:

- `src/layouts/ContentLayout.astro`

Recommended helper pattern:

- Keep prop names minimal but accurate. Prefer `authors?: string[]` at layout/header boundaries, and format to `Authors` display text in the header component.
- Update labels from `Author` to `Authors` where displaying the joined list.

For index-backed results:

- Continue accepting `row.author` from D1 as a display string and map it into existing card data as `author` or a new `authorsLabel` depending on least-disruptive implementation.
- For local collection entries, derive display from `entry.data.authors`.

### 7. Add Contributor Components

Add components:

- `src/components/ContributorCard.astro`
- `src/components/ContributorProfileHeader.astro` if useful; otherwise keep profile header inline for MVP.

Contributor card requirements:

- Name: `displayName ?? title`.
- Optional avatar.
- Optional `bioExcerpt`.
- Link to `/contributors/{id}`.
- Use existing DaisyUI/Tailwind/global utility styling and match `ContentCard` visual language.
- Use Astro `<Image>` only if avatar paths are importable/local assets; otherwise use plain `<img>` for string paths that may point to `/api/content-image` or public URLs. Do not overbuild avatar optimization in MVP.

### 8. Add Contributor Routes

Add `src/pages/contributors/index.astro`:

- Use `getCollection('contributors')`.
- Filter `status` to `publish` or `published`.
- Sort by display name/title.
- Render one unified grid of `ContributorCard`.
- Use `MainSiteLayout` or `SimpleContentLayout` with `GlobalNavigation` via existing layout.

Add `src/pages/contributors/[...slug].astro`:

- `getStaticPaths()` from published contributors.
- Render contributor markdown body with `render(entry)`.
- Show profile header/bio/socials.
- Build contribution list across public collections for `authors`/`contributors` matches:
  - MVP public collections: `lore`, `places`, `sentients`, `bestiary`, `flora`, `factions`, `systems`, `meta`.
  - Avoid campaign collections for MVP public profile listings unless the existing visibility/auth gating is explicitly handled.
- For `standard`: show all matching public entries.
- For `featured`: resolve only `featuredContributions` entries.
- Always include contributor search link.

Contribution card href issue:

- `ContentCard` currently builds `/${collection}/${id}` and works for canon/using collections.
- Reuse it for MVP public collections.
- If featured contributions can include campaign content later, add a route helper then.

Contributor search link MVP:

- Use `/search?q=${encodeURIComponent(displayNameOrSlug)}` or `/search?q=${encodeURIComponent(slug)}`.
- Prefer slug if content/index stores ids; prefer display name if search is more user-friendly.
- Since `/api/search` does not yet support `author`/`contributor` filters, label the link as broad search, e.g. `Search for more by or with this contributor`.

```bash

```


Potential caveat:

## Deferred Follow-Ups

### Image Attribution Validation

Later work in `scripts/content-sync/validate.mjs` or a focused validation module:

- Detect markdown image followed immediately by whitespace and `[[contributors/...]]` caption.
- Warn/fail when `artist` contributor frontmatter lacks matching image caption.
- Warn/fail when image caption credits an artist missing from `contributors` frontmatter.
- Resolve contributor ids against contributor entries.

### Contributor-Aware Search

Later D1/index work:

- Add `authors_json` and `contributors_json` or a separate contributor index table.
- Extend `/api/search` with `contributor=<slug>` filter.
- Update contributor profile links from broad search query to exact contributor search.

### Lightbox

Later UX work:

- Add single-image progressive lightbox around semantic image links.
- No gallery state in MVP.

## Risks

- If any content still uses `author`, the schema/build will fail after switching to required `authors`.
- Keeping `content_index.author` as a derived display string is intentionally pragmatic but not a full contributor-query model.
- Contributor profile listing across campaign collections is excluded in MVP to avoid leaking protected content.
- Active sync config is gitignored; contributor content will not sync until the local config includes the new mapping.

## Ready Criteria

Implementation is ready to begin when these assumptions are accepted:

1. `authors` is now canonical; no long-term `author` fallback.
2. Contributor markdown lives in the Obsidian vault under a `contributors` folder.
3. MVP contributor profiles list only public non-campaign collections.
4. Contributor profile search links can be broad `/search?q=...` links until exact contributor filters exist.
5. Image attribution validation and lightbox are deferred.
