# Sorcerer Spell Search and Filter UX Plan

## Goal
Implement the updated public spell API search/filter contract for the sorcerer spell browser, while replacing the current dominant 92-type landing-page grid with a more useful search-first experience.

## Recommended UX
Use a server-rendered GET form that submits to `/systems/gurps/resources/sorcerer-spells/all` and preserves filters in query parameters.

Primary controls should be visually prominent:

- Full-text search, `q`: primary input, labeled like `Search spell text`, with helper text that it searches spell name, description, and Archmagister's Counsel.
- Spell name prefix, `name`: secondary primary-row input, labeled like `Spell name starts with`.
- Spell type, `type`: enumerated select populated from `listSpellTypes()`.
- Submit and clear actions: `Search Spells` and `Clear Filters`.

Source lineage controls should be available but visually secondary:

- Put `sourceName` and `sourceType` inside a collapsed or low-emphasis `<details>` block labeled `Source filters` or `Advanced source filters`.
- Use smaller text and muted `text-ink-700`/`meta-muted` styling to indicate these are not the main discovery path.
- Source spell type is a distinct facet from active spell type. Do not populate the `sourceType` dropdown from `listSpellTypes()` unless the API/docs are changed to say those exact enumerations are identical.
- Because the OpenAPI contract documents `sourceType` as a filter but does not expose a separate source-spell-type listing endpoint, the implementation needs a distinct `sourceSpellTypes` option source. Preferred order:
  - Use a newly documented/source-controlled source-spell-type enum if one exists outside the inspected OpenAPI route list.
  - If no such enum exists, add a small local constant/data file for source spell type labels, clearly named as source lineage filter options, and treat it as UI metadata rather than a second spell-data source.
  - Do not derive source spell type options by fetching every spell page on each request; that would make the secondary filter too expensive and brittle.

Landing page changes:

- Keep the attribution block.
- Add a new `Find Spells` panel above browse links, using the same search/filter form.
- Replace the large type-button wall with a compact `Browse by type` area after the search panel.
- Recommended compact type treatment: a collapsed `<details>` section with small wrap chips or a compact multi-column list, linking each type to `/systems/gurps/resources/sorcerer-spells/all?type=...`.
- Keep the existing `/systems/gurps/resources/sorcerer-spells/[type]` route for backward compatibility and sitemap stability unless there is a separate cleanup task.

Results page changes:

- Render the same filter form at the top of `/all` and `/all/[page]`.
- Use the active filters to call `listSpells({ page, pageSize, q, name, type, sourceName, sourceType })`.
- Change result copy from always saying `alphabetical order` to conditional copy:
  - No filters: `Showing 1-100 of N sorcerer spells in alphabetical order.`
  - `q` present: `Showing 1-100 of N matching sorcerer spells.` and mention full-text relevance ordering if useful.
  - Filters present without `q`: `Showing 1-100 of N filtered sorcerer spells.`
- Add an empty-state message when no spells match, with a clear-filters link.
- Preserve all active filters in pagination links.

## UX Options Considered

1. Recommended: server-rendered search/filter form on landing and results pages.
   - Best fit for current Astro/Cloudflare server output.
   - No client framework, no new dependency, no browser-side API-origin coupling.
   - Shareable/bookmarkable URLs.
   - Works with rate-limit and service-unavailable behavior already modeled by the adapter.

2. Type-ahead/autocomplete using `/spell-suggestions`.
   - Useful later, but not recommended for this slice because it requires client-side behavior in the Using Aletheia/GURPS area and the handoff explicitly says to leave type-ahead for a later slice.

3. Separate advanced search page.
   - Avoids crowding the list page, but adds another route and makes the main spell page less useful.
   - Not recommended because the current problem is the main page being dominated by type browsing instead of search.

## Implementation Steps

1. Update the API adapter contract parity.
   - File: `src/adapters/public-spell-api.ts`.
   - Add `name: string` to `PublicSpellPage` because the OpenAPI response requires it.
   - Add `name?: string` to `ListSpellsParams`.
   - Pass `name: params.name` in the `/spells` query object.
   - Do not add `name` to `SuggestSpellsParams`; the OpenAPI suggestion route still uses `q` as spell-name prefix plus optional type/source filters.

2. Update adapter tests.
   - File: `src/adapters/public-spell-api.test.ts`.
   - Include `name` in the list response fixture.
   - Assert `/spells` query construction includes `name=...` with the documented parameter name.
   - Keep suggestion query tests unchanged except for fixture consistency if needed.

3. Add spell filter helpers.
   - File: `src/utils/spell-browser.ts`.
   - Add a `SpellSearchFilters` interface with `q`, `name`, `type`, `sourceName`, and `sourceType` as strings.
   - Add `getSpellSearchFilters(searchParams: URLSearchParams): SpellSearchFilters` that trims values and returns empty strings for missing values.
   - Add `hasSpellSearchFilters(filters)`.
   - Add a URL builder such as `getSpellListPageHref(page, filters?)` or a new helper that preserves non-empty filters when building `/all` and `/all/[page]` links.
   - Update `createEmptySpellPage()` to include `name: ''`.

4. Create a reusable Astro filter form component.
   - Suggested file: `src/components/spells/SpellSearchForm.astro`.
   - Props: `filters`, `spellTypes`, `sourceSpellTypes`, and optional `resultTotal`/`compact` if needed.
   - Render a GET form with `action="/systems/gurps/resources/sorcerer-spells/all"`.
   - Primary row: `q`, `name`, `type`.
   - Secondary source filters: `<details class="mt-4 text-sm text-ink-700">` with `sourceName` and `sourceType`.
   - Use semantic DaisyUI/Tailwind/project classes already present: `ui-surface`, `ui-button`, `bg-base-100`, `text-ink-*`, `border-base-300`, `focus-ring` if appropriate.
   - Ensure labels are explicit for accessibility.
   - The active spell type select should include an `Any spell type` option with empty value and use `spellTypes` from `listSpellTypes()`.
   - The source spell type select should include an `Any source type` option with empty value and use the distinct `sourceSpellTypes` option source.

5. Update the landing page.
   - File: `src/pages/systems/gurps/resources/sorcerer-spells/index.astro`.
   - Import `SpellSearchForm`.
   - Keep fetching `listSpellTypes()` and the one-item spell page for total count.
   - Load distinct source spell type options from the agreed option source; do not use active spell types for this dropdown.
   - Replace the `Browse Spells` and large `Spell Types` grid with:
     - `Find Spells` panel containing `SpellSearchForm` with empty filters.
     - Compact browse links: `All Spells (Alphabetical)` plus collapsed/smaller type chips.
   - Type chips should preferably link to `/systems/gurps/resources/sorcerer-spells/all?type=<encoded type>` so users land in the unified filterable results UI.

6. Update list pages to read and apply filters.
   - Files:
     - `src/pages/systems/gurps/resources/sorcerer-spells/all/index.astro`
     - `src/pages/systems/gurps/resources/sorcerer-spells/all/[page].astro`
   - Fetch `listSpellTypes()` for the active spell type dropdown alongside `listSpells()`.
   - Load distinct source spell type options for the secondary source type dropdown.
   - Parse filters from `Astro.url.searchParams`.
   - Pass filters into `listSpells()`.
   - Preserve filters in invalid-page and page-clamping redirects.
   - Update title/description when filters are active, e.g. `Search Spells` instead of always `All Spells (Alphabetical)`.

7. Update `SpellAlphabeticalIndex` and pagination.
   - Files:
     - `src/components/spells/SpellAlphabeticalIndex.astro`
     - `src/components/spells/SpellPaginationNav.astro`
   - Add `filters` prop to `SpellAlphabeticalIndex`.
   - Render `SpellSearchForm` above the result summary or have the page render it above the component; prefer the page-level render if it keeps `SpellAlphabeticalIndex` focused on results.
   - Add `filters` prop to `SpellPaginationNav` and use the updated URL helper so query params persist.
   - Adjust summary text and empty-state behavior based on `hasSpellSearchFilters(filters)`.

8. Leave type-specific route mostly unchanged.
   - File: `src/pages/systems/gurps/resources/sorcerer-spells/[type].astro`.
   - Do not refactor this route in the main slice unless needed.
   - It remains useful for existing links and sitemap entries, though the main landing page can start routing users to the unified filtered `/all?type=...` experience.

9. Verify.
   - Run `pnpm test -- public-spell-api` or the closest supported Vitest filter for adapter tests.
   - Run `pnpm build` to catch Astro/TypeScript/template issues.
   - Manually check URLs:
     - `/systems/gurps/resources/sorcerer-spells`
     - `/systems/gurps/resources/sorcerer-spells/all?q=hidden+steel`
     - `/systems/gurps/resources/sorcerer-spells/all?name=Abs&type=Adventurer+Spells`
      - `/systems/gurps/resources/sorcerer-spells/all?sourceName=Absorb&sourceType=<known-source-spell-type>`
      - pagination with filters, e.g. `/all/2?q=ward&type=Air+Spells`.

## Notes and Constraints

- Use `pnpm` only.
- Add no dependencies.
- Keep implementation Astro-native and server-rendered.
- Do not introduce React/Vue/Svelte islands for this slice.
- Keep source lineage controls present but visually secondary.
- Source spell type and active spell type are different enumerations; keep their option sources, labels, and helper text distinct.
- The public API is external/read-only; this repo should consume it through `src/adapters/public-spell-api.ts`, not implement D1 access directly.
