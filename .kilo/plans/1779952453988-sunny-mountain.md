## Scope

In scope:

1. Browser-local temporary spell list keyed by `spell_id`.
2. Vanilla TypeScript client behavior only; no framework island and no new dependency.
3. Add/remove controls on spell browsing rows and spell detail dialogs.
4. Dedicated saved spell-list page for review and printing.
5. Clear UX copy that persistence is browser/device local only.
6. Tests for storage parsing/state behavior where practical.
7. `pnpm test` and `pnpm build` verification.

Out of scope:

1. Account-backed saved spell lists.
2. Cross-device sync.
3. D1 persistence.
4. Multiple named lists.
5. Sharing/import/export.
6. Spell CRUD/admin surfaces.
7. Search S2 relevance/snippet/highlighting work.
8. Backend changes to the public spell API unless implementation discovers a blocker.

## Implementation Plan

### Phase 1 - Local Storage Foundation

Add a small utility module under `src/utils/` for spell-list state.

Expected shape:

- Export a stable storage key, for example `woa:sorcerer-spell-list:v1`.
- Store only an array of `spell_id` strings.
- Normalize IDs by trimming and dropping empty/non-string values.
- Deduplicate while preserving insertion order.
- Treat add/remove as idempotent.
- Catch malformed JSON and storage access errors.
- Return safe empty state when storage is unavailable.

Likely file:

- `src/utils/spell-list-storage.ts`

Likely tests:

- `src/utils/spell-list-storage.test.ts`

Test cases:

1. Empty/missing storage returns an empty list.
2. Malformed JSON returns an empty list without throwing.
3. Non-array JSON returns an empty list.
4. Mixed invalid entries are normalized away.
5. Duplicate IDs collapse to one value.
6. Add is idempotent.
7. Remove is idempotent.
8. Storage write failure does not break read/update callers.

### Phase 2 - Shared Client UI Behavior

Add a tiny browser script that wires spell-list buttons on rendered pages.

Recommended approach:

- Use data attributes instead of inline per-button state logic.
- Buttons expose `data-spell-list-toggle` and `data-spell-id`.
- Optional labels/counters expose stable data attributes.
- On load, read saved IDs and update all controls on the page.
- On click, toggle the ID, persist state, and update all matching controls.
- Dispatch/listen for a small custom event if needed so multiple controls for the same spell stay synchronized.

Likely file:

- `src/components/spells/SpellListClientScript.astro` or inline script in a small component imported by spell pages/components.

UI behavior:

- Not saved: button label `Add to Spell List`.
- Saved: button label `Remove from Spell List`.
- Use `aria-pressed` to reflect current state.
- Keep controls usable when JavaScript loads after initial HTML.
- If storage is unavailable, controls should fail softly rather than break spell browsing.

### Phase 3 - Add Controls To Existing Spell Surfaces

Update spell rendering surfaces to include controls.

Likely files:

- `src/components/spells/SpellAlphabeticalIndex.astro`
- `src/components/spells/SpellDetailsDialog.astro`
- `src/pages/systems/gurps/resources/sorcerer-spells/index.astro` if type/index landing needs a saved-list link or count
- `src/pages/systems/gurps/resources/sorcerer-spells/all/index.astro`
- `src/pages/systems/gurps/resources/sorcerer-spells/all/[page].astro`

Table integration:

- Add an `Action` column or extend the `Details` column with two compact actions.
- Keep mobile/table overflow readable.
- Avoid making the table substantially wider than necessary.

Dialog integration:

- Add the same toggle control near the dialog footer or header.
- Controls for the same spell must stay synchronized between row and dialog.

Navigation affordance:

- Add a link to the saved spell-list page near the spell search/filter surface.
- Include concise persistence note near the link or saved-list page intro: `Your spell list is stored only in this browser on this device.`

### Phase 4 - Saved Spell-List Page

Add a dedicated route for reviewing and printing selected spells.

Recommended route:

- `src/pages/systems/gurps/resources/sorcerer-spells/list.astro`

Reasoning:

- Keeps the saved list under the existing Sorcerer Spells route family.
- Avoids conflicting with `/all` pagination/search routes.
- Gives users a stable URL to return to from browse pages.

Rendering model:

- Server-render the page shell and include client script to hydrate selected IDs from localStorage.
- Because localStorage is client-only, the saved-list contents must be populated in the browser after load.
- Fetch spell details for saved IDs using the existing public API endpoint shape `/api/v1/spells/{spell_id}` or `PUBLIC_SPELL_API_BASE` equivalent if needed in browser code.
- If browser-side API base handling is awkward, prefer a same-origin relative public API path only if that endpoint exists in the deployed/public site; verify before implementing.

Saved-list behavior:

- Empty state explains no spells are saved and links back to browse all spells.
- Loading state is brief and accessible.
- Missing/stale IDs are reported in a small warning area and can be removed.
- Valid spells render in a stable, scan-friendly order.
- Recommended order: preserve user add order for tabletop list-building. If that proves confusing, sort by spell name only on the print page, but keep this as an implementation note rather than a larger feature.

Print requirements:

- Hide navigation/action controls in print via print utility classes or component CSS.
- Show spell name, types, cost, casting roll, range, duration, description, statistics, and Archmagister's Counsel where useful.
- Avoid modal/dialog UI for printed output.
- Keep entries readable across page breaks.

### Phase 5 - Verification

Run:

1. `pnpm test`
2. `pnpm build`

Manual verification checklist:

1. Add a spell from the all-spells table.
2. Open its detail dialog and confirm the dialog control reflects saved state.
3. Remove the spell from the dialog and confirm the table control updates.
4. Reload the page and confirm saved state persists.
5. Navigate to `/systems/gurps/resources/sorcerer-spells/list` and confirm saved spells render.
6. Remove a spell from the saved-list page and confirm it disappears without page breakage.
7. Clear localStorage and confirm the saved-list page shows the empty state.
8. Manually corrupt the storage key and confirm the UI recovers safely.
9. Open print preview and confirm action controls/navigation are hidden or de-emphasized and spell entries are readable.

## Risks And Mitigations

- Risk: localStorage is unavailable or blocked.
  Mitigation: catch storage errors and keep spell browsing functional.

- Risk: saved IDs become stale after API data changes.
  Mitigation: saved-list page reports missing IDs and offers removal.

- Risk: client page needs to fetch many spell details.
  Mitigation: keep this first version simple; if performance is poor, add batching only if the public API supports it or after a concrete need is proven.

- Risk: add/remove controls clutter the spell table.
  Mitigation: keep controls compact and reuse existing `ui-button` styles.

- Risk: saved-list route implies durable/account-backed persistence.
  Mitigation: clear page copy states browser/device-local persistence.

## Acceptance Criteria

1. Visitors can add/remove spells from existing spell browsing/detail surfaces.
2. Saved spell IDs persist across reloads in the same browser.
3. Duplicate saves are impossible or collapse idempotently.
4. Malformed storage and stale IDs do not break the UI.
5. Dedicated saved-list page works for empty, one-spell, and many-spell states.
6. Print view is usable and omits unnecessary controls.
7. Implementation remains Astro-native and vanilla TypeScript.
8. No new dependencies are added.
9. `pnpm test` passes.
10. `pnpm build` passes.
