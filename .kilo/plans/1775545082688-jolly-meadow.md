## Implementation Plan

### A) Wrangler staging config hardening
1. Diff top-level Wrangler bindings against `env.staging` for parity-sensitive entries (especially storage bindings used by D1/sync/operator lanes).
2. If any required binding parity is missing, add the explicit staging equivalent in `wrangler.jsonc`.
3. Re-run a representative staging command (`pnpm db:migrate:plan:staging:dry-run` and one `wrangler d1 execute ... --env staging`) to verify warning-free output.

Expected outcome:
- No staging warning noise tied to missing environment binding parity.

### B) Migration parser hardening
1. Tighten `queryNumeric()` behavior to distinguish:
   - successful query with numeric row,
   - successful query with non-numeric payload,
   - parse failure/no-row condition,
   - Wrangler command failure.
2. Prefer fail-closed behavior for conflict-detection queries: if critical query output is unreadable, return an explicit error path instead of defaulting to `0`.
3. Keep existing mixed-output JSON extraction, but make parsing diagnostics explicit in returned errors to aid operator troubleshooting.
4. Remove dead/unused parsing helpers if no longer needed.

Expected outcome:
- Dry-run cannot silently pass/fail due to warning-polluted output parsing.

### C) Tests
1. Add targeted tests for parser extraction and numeric query interpretation using representative Wrangler mixed output (human logs + JSON payload).
2. Add a regression case that previously produced false `schema_object_conflict` behavior under noisy output conditions.
3. Ensure test suite remains green with `pnpm test`.

Expected outcome:
- Parser robustness is verified and protected from regressions.

### D) Documentation close-out (minimal, in-scope)
