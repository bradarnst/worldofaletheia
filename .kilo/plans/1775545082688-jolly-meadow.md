# Follow-On Plan - Reference and Calendar Workstream Rebaseline

## Rebaseline Notes
- Prior operator-tooling Step 1 is complete and no longer the active sequencing concern.
- The earlier remote parity automation decision remains intentionally deferred for at least a week.
- Calendar work remains active, but the newly accepted Reference layer now changes the order of work.
- Reference domain and route rehome must happen before richer calendar UI work continues.

## New Ordered Queue

### Step 1 - Establish the Reference layer and `/references/*` route namespace
- Scope:
  - Introduce Reference as the fourth top-level information architecture layer.
  - Move Calendar and Timeline under `/references/calendar` and `/references/timeline`.
  - Create `/references` and `/references/maps` as the stable hub and map surface.
  - Record the architecture decision and produce code handoff documentation for nav/layout/route migration.
- Why first:
  - Prevents ongoing one-off nav growth and gives Calendar, Timeline, and Maps a coherent home.
  - Reduces rework before more calendar UI, maps, and discoverability work land.
- Exit criteria:
  - One accepted ADR locks the Reference domain and route namespace.
  - A code-facing HLD exists for nav/layout/route migration.
  - Active planning docs reflect `/references/*` as canonical routes.

### Step 2 - Revise the civic Leap Day rule on the rehomed calendar surface
- Scope:
  - Keep Leap Day outside month numbering but inside the weekday cycle.
  - Update calendar specification, engine expectations, and API/UI contracts against `/references/calendar`.
- Why next:
  - Leap Day correctness should land before richer calendar UI and eclipse enrichment expand the implementation surface.
- Exit criteria:
  - One accepted written decision locks the revised civic rule.
  - Calendar handoff/spec docs reflect the new weekday-bearing Leap Day model under the Reference route family.

### Step 3 - Build the richer calendar consumer and UI/UX uplift
- Scope:
  - Improve `/references/calendar` as the primary consumer of the shared calendar engine and `/api/calendar/*` surface.
  - Add clearer previous/next navigation, visual moon icons, Leap Day visibility in month/year UI, and a first-class selected-day agenda/detail pattern.
  - Keep events visible by density: light in month view, richer in week view, full agenda on day detail.
  - Close out the tranche by removing temporary explanatory helper copy and non-canonical visible labels that leak implementation detail.
- Why next:
  - Once route placement and Leap Day rules are fixed, the UI can be designed against a stable canonical model.
- Exit criteria:
  - `/references/calendar` exposes materially better navigation and day-detail UX.
  - Leap Day is clearly represented in month, week, and year views.
  - No user-facing explanatory helper copy remains where the UI should stand on its own.
  - Passing `pnpm test` and `pnpm build` remain the implementation bar.

### Step 4 - Add eclipse schedule enrichment to the calendar
- Scope:
  - Implement deterministic eclipse-season/eclipsedate math inside the shared calendar engine.
  - Surface eclipses in calendar/API outputs and calendar UI.
  - Defer planetary path/coverage and map-aware visibility to a later cartographic step.
- Why after Step 3:
  - Eclipse data should land on top of the corrected Leap Day model and improved Reference calendar presentation layer.
- Exit criteria:
  - Calendar surfaces show computed eclipse occurrences without needing authored content entries.
  - No geographic path/coverage claims are introduced yet.

### Step 5 - Expand calendar interaction and agenda modes
- Scope:
  - Add a stronger selection model across day, week, month, and year surfaces.
  - Support summary panels driven by selected granularity rather than day-only assumptions.
  - Add explicit secondary actions for deeper navigation instead of relying on double-click behavior.
  - Introduce separate agenda views for day, week, month, and year.
- Why later:
  - This is a real interaction-model expansion and should follow the current calendar presentation and eclipse foundation work.
- Exit criteria:
  - Selection and summary behavior are coherent across calendar views.
  - Dedicated agenda modes exist for day, week, month, and year.
  - Navigation affordances make primary selection vs secondary “go to/open” actions clear.

### Step 6 - Make an explicit decision on remote parity automation
- Scope:
  - Decide between continued manual/operator-run parity checks and a secret-backed CI lane for remote verification.
  - If automated, define ownership, failure policy, and required secrets before workflow changes.
- Why later:
  - Still valuable, but intentionally delayed while Reference/calendar work is the active product-facing focus.
- Exit criteria:
  - One written policy (manual vs automated) plus any required runbook update.

### Step 7 - Run a doc-priority cleanup pass to reduce drift
- Scope:
  - Update active planning docs/checklists still carrying stale/open markers from pre-reference-rebaseline phases.
  - Mark superseded content clearly and keep one authoritative current-priorities source.
- Why later:
  - Prevents plan drift after the Reference and calendar sequence settles.
- Exit criteria:
  - Open-item counts and priority order reflect current reality.

### Step 8 - Then schedule the next structural workstream (recommended: Option 3 authz unification)
- Scope:
  - Start with ADR + migration strategy for unified `campaign_memberships` roles (`plans/option-3-unified-membership-role-upgrade-todo.md`).
- Why later:
  - Valuable simplification, but lower priority than Reference/calendar correctness and near-term UX enrichment.
