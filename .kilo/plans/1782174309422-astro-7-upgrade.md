# Astro 6 → 7 Upgrade Plan

## Status
- Date: 2026-06-24
- Status: Draft — ready for implementation
- Source request: Upgrade to Astro 7 (Advanced Routing + faster builds)
- Reference: https://docs.astro.build/en/guides/upgrade-to/v7/ , https://astro.build/blog/astro-7/

## Current State
- `astro` 6.1.9
- `@astrojs/cloudflare` 13.2.1 (peer-depends on `astro ^6.0.0`)
- `astro-seo` ^1.1.0
- `@tailwindcss/vite` ^4.2.4 (Vite plugin), `tailwindcss` ^4.2.4
- `@aws-sdk/client-s3` ^3.1037.0 (runtime dep in custom R2 loader)
- `astro.config.mjs`: `output: 'server'`, no integrations, no `markdown.*` config (default remark/rehype), no experimental flags, no `compressHTML` override, no MDX, no `src/fetch.ts`
- Custom content loader `src/lib/r2-content-loader.mjs` and sync helper `scripts/content-sync/cloud-content-metadata.mjs` reach into the pnpm store to import `parseFrontmatter` from `@astrojs/markdown-remark` (a package no longer installed by default in Astro 7).

## Goal
Upgrade to Astro 7 with `@astrojs/cloudflare` 14.x, replacing the fragile pnpm-store frontmatter hack with a normal Astro-native import.

## Knock-on Upgrades (required)
1. `astro`: 6.1.9 → 7.x
2. `@astrojs/cloudflare`: 13.2.1 → 14.x (13.x is incompatible — it peer-depends on `astro ^6.0.0`; v14.0.0 shipped with Astro 7 and upgrades to Vite 8)
3. Add `@astrojs/internal-helpers` as an explicit dependency (Astro-org internal package; required for resolvability under pnpm strict isolation)

## Core Code Change (the one real blocker)
`@astrojs/markdown-remark` is no longer installed by default in Astro 7 (Sätteri is the new Markdown processor). The pnpm-store path scans will fail. Replace with a normal import of the exact function Astro 7's own content layer uses.

- `src/lib/r2-content-loader.mjs`:
  - Remove `getParseFrontmatter()` (lines 17-42) and the pnpm-store scan.
  - Replace with: `import { parseFrontmatter } from '@astrojs/internal-helpers/frontmatter';`
  - Update the `load()` usage (line 233, 246) to call the imported `parseFrontmatter` directly. Loader flow is unchanged: parse frontmatter to get `{ frontmatter, content }`, enrich/sanitize metadata, validate via `context.parseData`, image-preprocess `content`, then `context.renderMarkdown`.
- `scripts/content-sync/cloud-content-metadata.mjs`:
  - Apply the same replacement to its duplicated `getParseFrontmatter()` helper.
- Keep `@astrojs/markdown-remark` OUT of dependencies (do not add it back).

Rationale: the loader needs standalone frontmatter parsing to separate the body before rendering and to derive metadata/routing, so it cannot rely on `renderMarkdown().metadata.frontmatter` alone. `@astrojs/internal-helpers/frontmatter::parseFrontmatter` is the same function `@astrojs/markdown-remark` re-exported and that Astro 7's `content-layer.ts` imports directly.

Caveat to acknowledge: `@astrojs/internal-helpers` is an internal helper, not a formally-documented public Astro API. Accepted as the lowest-risk Astro-native option.

## Config / No-Change Items (verify, likely clean)
- No experimental flags to remove (config has none).
- No `@astrojs/db`, no `getContainerRenderer`, no `astro:transitions` internals, no `createAnimationScope` usage (grep confirmed none).
- No MDX files in repo.
- No `src/fetch.ts` collision (Advanced Routing's reserved name is free; without the file Astro behaves as today).
- No `markdown.remarkPlugins` / `rehypePlugins` in config — Sätteri default is a clean swap for rendering.

## Validation-Only Risks (no upfront code; surface via build + visual pass)
1. Rust compiler strictness: unclosed tags / invalid HTML nesting now error or render differently across many `.astro` files. Fix as the build reports them.
2. `compressHTML: 'jsx'` is the new default — whitespace between inline elements collapses (JSX-style). Inspect nav bars, tag chips, badges, inline icons. If spacing regresses, add `compressHTML: true` to `astro.config.mjs` to preserve prior behavior; otherwise fix per-element with `{' '}`.
3. Sätteri Markdown processor — spot-check Obsidian-authored content (GFM, smartypants, heading IDs, wikilinks now built-in). No plugins to port, but compare a few rendered pages before/after.
4. `astro-seo@1.1.0` — verify it installs and works under Astro 7 (expect possible peer-dep warning; it is a component, likely fine).
5. CSS cosmetic differences — named colors may serialize to hex, `url()` quoting may flip. Only matters if tests assert exact CSS strings (none found).
6. Vite 8 / Rolldown — `@tailwindcss/vite` v4 compatibility; `@aws-sdk/client-s3` is runtime-only. Validate via build.

## Notable Cons of Upgrading
- Adds one internal-package dependency (`@astrojs/internal-helpers`, not a documented public API).
- Likely one-time fixes for HTML/Markdown surfaced by the stricter Rust compiler and Sätteri.
- One visual regression pass required for whitespace behavior.
- `compressHTML` may force a global config fallback if inline spacing regresses broadly.

## Validation Steps (ordered)
1. `pnpm install` after bumping `astro`, `@astrojs/cloudflare`, adding `@astrojs/internal-helpers`.
2. `pnpm exec tsc` — typecheck.
3. `pnpm test` — existing vitest suite (loader/repo/scripts/migration tests).
4. `pnpm build` — default local build; watch for Rust compiler HTML errors and Sätteri rendering issues.
5. `pnpm dev:cf:build` (`CONTENT_SOURCE_MODE=cloud CONTENT_LOADER_D1_MODE=local astro build`) — the cloud-content build path that exercises the frontmatter change; confirm `parseFrontmatter` import resolves and R2 content loads.
6. Local `astro dev` + `astro preview` visual pass across the four themes (`savanna-days`, `lake-days`, `savanna-nights`, `jungle-nights`), focusing on nav/chips/badges/inline spacing and a few Obsidian-authored content pages.
7. Staging build + deploy (`pnpm deploy:staging`) before production.
8. Production deploy (`pnpm deploy:prod`) only after staging verifies.

## Rollback
If blocking issues surface, revert `package.json` versions to `astro@6.1.9` + `@astrojs/cloudflare@13.2.1`, restore the `getParseFrontmatter()` pnpm-store helper, and `pnpm install`. The frontmatter change is isolated and reversible.

## Out of Scope
- Adopting Advanced Routing (`src/fetch.ts`) to reorder auth before Actions — a separate follow-up; not required for this upgrade. Noted as a future benefit the upgrade unlocks.
- Adopting route caching / CDN cache providers (Cloudflare cache provider is private beta — do not enable).
- Refactoring the loader to use `context.renderMarkdown().metadata.frontmatter` — intentionally not done; standalone parsing is needed by the current flow.
- Any D1 migration (none required for Astro upgrade).
