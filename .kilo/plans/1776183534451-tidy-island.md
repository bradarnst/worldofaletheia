# UI Revamp Completion Plan

## Goal
Complete the production UI revamp using the approved wireframe direction and finish Daisy extraction without breaking Astro-native boundaries.

## Current State (Verified)
- Wireframes are approved and captured in `plans/ui-component-and-shell-spec-2026-04-15.md`.
- Prototype routes were removed.
- New production-level shells already exist for:
  - global header/nav (`src/components/GlobalNavigation.astro`)
  - homepage hero/support structure (`src/pages/index.astro`)
  - collection shell/filter/cards (`src/components/DiscoveryCollectionView.astro`, `src/components/site/CollectionHeroShell.astro`, `src/components/site/CollectionFilterBar.astro`, `src/components/ContentCard.astro`)
  - article context header (`src/components/site/ArticleContextHeader.astro`)
  - compact footer (`src/components/site/CompactFooter.astro`)
- Daisy is still active globally (`src/styles/global.css` includes `@plugin "daisyui"`; `package.json` still includes `daisyui`).
- Daisy class usage remains widespread in Campaigns and Reference surfaces (buttons/cards/badges/inputs/selects).

## What Is Left To Finish The UI Revamp

### 1) Lock Phase 1 Foundations (Token + Theme Ownership)
1. Audit `src/styles/global.css` token definitions against approved shell spec and remove ambiguity between Daisy semantics and site-owned tokens.
2. Ensure typography/spacing/radius tokens are the source of truth for new shells.
3. Define explicit compatibility policy for remaining Daisy classes during migration.

### 2) Complete Production Rollout Of Approved Shell Direction
1. Bring all collection-like pages to the same approved shell behavior (including `src/pages/meta/index.astro`, which still uses a legacy list pattern).
2. Ensure homepage shell remains aligned to approved rules (centered hero copy, no framed hero copy card, curated support sections).
3. Verify article surfaces across all domains use the approved structure consistently (shared content column alignment, subtle context block, compact metadata hierarchy).
4. Normalize card/filter/footer/header treatments where old rounded or legacy patterns still appear.

### 3) Finish Dense Utility Surfaces (Phase 5)
1. Migrate Reference calendar/timeline controls from Daisy form/button/badge primitives to owned styles while preserving behavior.
2. Migrate Campaigns index/session surfaces from Daisy card/button/badge patterns to owned component language.
3. Decide and prototype the Reference sidebar pilot (explicit outstanding dependency in the handoff plan).

### 4) Replace Placeholder Glyphs With Real Icon Mapping (Phase 6)
1. Keep `SiteGlyph.astro` as the semantic entry point, but swap internals from initials placeholder to real SVG mappings.
2. Deliver top-level/domain and collection icons first.
3. Roll out type-enum icon backlog from `src/lib/content-types.ts` in progressive slices.

### 5) Resolve Remaining Approved-But-Unfinalized Visual Inputs
1. Final hero height values by breakpoint.
2. Final collection blur/tint/overlay numeric ranges.
3. Final collection background image assignment strategy (fixed vs rotating per collection).

### 6) Daisy Decommissioning (Phase 7)
1. Run full Daisy usage audit in `src/` and remove remaining Daisy contract classes route-by-route.
2. Remove Daisy compatibility aliases once no routes depend on them.
3. Remove `@plugin "daisyui"` from global CSS and remove `daisyui` dependency from `package.json`.
4. Perform final regression pass across light/dark themes, desktop/mobile, and route families.

## Recommended Execution Order
1. Foundation stabilization (tokens/theme/radius scale)
2. Campaigns + Reference migration away from Daisy primitives (largest remaining surface area)
3. Icon mapping rollout (top-level first)
4. Final visual tuning pass (hero heights, background values, image assignments)
5. Daisy removal + regression

## Definition Of Done For UI Revamp
- Approved shell direction is applied consistently across Homepage, Collection, Article, Campaigns, and Reference surfaces.
- Placeholder glyph initials are replaced with real semantic SVG mapping.
- Daisy classes are no longer required for active UI rendering.
- Daisy plugin/dependency is removed.
- Final QA pass confirms responsive behavior, theme parity, accessibility basics, and no critical visual regressions.
