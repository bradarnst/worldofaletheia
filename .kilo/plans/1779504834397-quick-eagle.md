## Implementation Notes For ADR-0021

Use MADR-style sections consistent with existing ADRs:

- Title: `External Admin Capability Boundary`
- Status: `Accepted` or `Proposed`, depending on implementation preference.
- Context and Problem Statement: admin capability needs exist, but this main site should remain public-site/API-consuming and Astro-native; admin CRUD/consoles live in `woa-admin` or related external projects.
- Decision Drivers: keep main site scope small, avoid privileged UI/runtime complexity, preserve external API boundary, support richer admin capabilities elsewhere, document handoff requirements clearly.
- Considered Options:
  - build admin dashboards in this repo
  - keep admin work external and consume APIs/artifacts from external projects
  - decide case-by-case with no default boundary
- Decision Outcome: external by default.
- Consequences:
  - Positive: clearer scope, safer public site, better separation of privileged admin concerns.
  - Negative: cross-project coordination and API contract dependency.
  - Neutral: main site may still have lightweight public forms and may consume external APIs such as spell search/query endpoints.
