# Plan: Move Sorcerer Spell Reads to Public D1-Backed API

## Goal
Refactor main-site sorcerer spell reads away from local JSON artifacts and through a minimal server-side adapter for the public spell API documented in `spells-move-to-D1-handoff/public-spell-read-api.openapi.yaml`.

Authoritative contract: `spells-move-to-D1-handoff/public-spell-read-api.openapi.yaml`.
Canonical production base: `https://worldofaletheia.com/api/v1/*`.
Main-site rule: no direct D1 access for spell reads.

## Existing Read Paths To Change
- `src/pages/systems/gurps/resources/sorcerer-spells/index.astro`
  - Currently imports `@data/spells/spells-raw.json` and `@data/spells/spell-types.json`.
  - Needs API-backed total spell count and spell type list.
- `src/pages/systems/gurps/resources/sorcerer-spells/[type].astro`
  - Currently imports spell JSON and filters by `spell.spell_type`.
  - Needs API-backed spell type lookup and `GET /api/v1/spells?type=...` reads.
- `src/pages/systems/gurps/resources/sorcerer-spells/all/index.astro`
  - Currently imports JSON, sorts locally, and paginates locally.
  - Needs `GET /api/v1/spells?page=1&pageSize=...`.
- `src/pages/systems/gurps/resources/sorcerer-spells/all/[page].astro`
  - Currently imports JSON for static path generation, sorting, pagination, and invalid-page redirect.
  - Needs runtime API pagination and contract page clamping behavior.
- `src/pages/sitemap.xml.ts`
  - Currently imports spell JSON and spell types to generate spell URLs.
  - Needs API-backed page count and spell type list.
- `src/components/spells/SpellAlphabeticalIndex.astro`
  - Consumes old `SpellRecord` shape, including singular `spell_type` and `keywords?: string`.
  - Needs contract `PublicSpell` shape with `spell_types: string[]`, `keywords: string[]`, `spell_id`, `archmagisters_counsel`, and `source_lineage` available.
- `src/components/spells/SpellDetailsDialog.astro`
  - Consumes old `SpellRecord` shape.
  - Needs contract fields and display rules for arrays/new fields.
- `src/utils/spell-browser.ts`
  - Defines the old local record shape, local sorting, and local pagination helpers.
  - Needs to become UI/helper-only plus shared public API types, or be split so API contract types live with the adapter.

No current `/src/pages/api/**` route reads spell data. No current tests cover spell reads.

## Implementation Plan

1. Add a minimal server-side adapter module.
   - Create `src/adapters/public-spell-api.ts` because this is a real external public HTTP API boundary, satisfying the repo guardrail for using `src/adapters/`.
   - Define explicit TypeScript interfaces matching the OpenAPI schemas:
     - `PublicSpell`
     - `PublicSpellSuggestion`
     - `PublicSpellPage`
     - `PublicSpellErrorResponse`
     - query parameter types for list/suggestion reads.
   - Preserve snake_case contract field names exactly.
   - Use `fetch()` against `https://worldofaletheia.com/api/v1` by default.
   - Implement these functions:
     - `listSpellTypes(): Promise<string[]>`
     - `listSpells(params): Promise<PublicSpellPage>`
     - `getSpellById(spellId): Promise<PublicSpell>`
     - `suggestSpells(params): Promise<PublicSpellSuggestion[]>`
   - URL-encode query parameters with contract names: `page`, `pageSize`, `q`, `type`, `sourceName`, `sourceType`, `limit`.
   - Do not locally duplicate normalization semantics beyond omitting empty/undefined params where safe; let the API normalize pages, page sizes, whitespace, casing, wildcard escaping, and clamping.

2. Add explicit error handling in the adapter.
   - Parse documented JSON error shape `{ error, message }` when present.
   - Preserve response `status`, `error`, `message`, and optional `Retry-After`.
   - Model failures as a typed error class or discriminated result; prefer a small `PublicSpellApiError` class if it keeps page code readable.
   - Required behavior:
     - `400` from detail lookup is an invalid request and should not be retried.
     - `404` from detail lookup is not found and should map to not-found UI/response where detail pages are introduced or where direct lookup is used.
     - `429` and `503` are retryable service failures; keep `Retry-After` available for UI/logging and avoid falling back to JSON/D1.
     - Unexpected non-2xx responses should be surfaced as API failures, not silently swallowed.
     - Network/parse failures should become service-unavailable-style adapter errors for the page boundary.

3. Migrate page rendering to API-backed reads.
   - Convert sorcerer spell pages from `prerender = true` to request-time server rendering, or remove the explicit prerender flag where runtime API reads are required.
   - `sorcerer-spells/index.astro`:
     - call `listSpellTypes()`.
     - call `listSpells({ page: 1, pageSize: 1 })` only for `total`.
     - render existing type navigation with slugified API type labels.
   - `sorcerer-spells/all/index.astro`:
     - call `listSpells({ page: 1, pageSize: SPELLS_PAGE_SIZE })`.
     - pass contract page metadata to the existing index component.
   - `sorcerer-spells/all/[page].astro`:
     - remove static path generation based on local JSON.
     - parse page param and redirect invalid/non-integer/below-2 pages to page 1.
     - call `listSpells({ page, pageSize: SPELLS_PAGE_SIZE })`.
     - preserve API clamping for pages above `totalPages`; if current URL is above final page and `totalPages > 0`, redirect to the canonical clamped page URL.
   - `sorcerer-spells/[type].astro`:
     - remove static paths based on local JSON.
     - call `listSpellTypes()` to resolve slug to canonical label.
     - if no slug match, return/redirect as not found according to current site conventions.
     - call `listSpells({ type: canonicalType, page: 1, pageSize: 100 })`.
     - If more than 100 spells can exist in a type, either fetch subsequent pages server-side or add pagination for type pages. Prefer fetching all pages only if current type pages are intended to remain full lists.
   - `sitemap.xml.ts`:
     - call `listSpellTypes()` and `listSpells({ page: 1, pageSize: SPELLS_PAGE_SIZE })`.
     - generate the spell overview, all-spells pages from `totalPages`, and type paths from API type labels.
     - If the spell API is 429/503, return a controlled 503 XML/text response rather than generating stale JSON-based URLs.

4. Update spell UI components for contract fields.
   - Update `SpellAlphabeticalIndex.astro` to accept `PublicSpell[]` or a UI alias based on it.
   - Display type as `spell.spell_types.join(', ')` instead of `spell.spell_type`.
   - Update keyword rendering to accept `string[]`; treat empty arrays as no keywords.
   - Update modal to include documented new fields that are useful to users:
     - `archmagisters_counsel`
     - `source_lineage.source_spell_name`
     - `source_lineage.source_spell_types`
   - Keep existing card/table visual style; avoid broad redesign.
   - Update modal IDs to prefer stable `spell_id` instead of name/index when available, reducing collisions after spell names change.

5. Keep helpers minimal and localized.
   - Retain UI helpers in `src/utils/spell-browser.ts`:
     - `SPELLS_PAGE_SIZE`, set to a value supported by the contract maximum. Current value `100` is acceptable because contract clamps max pageSize to 100.
     - `slugifySpellType()` for path compatibility.
     - `formatSpellField()`.
     - `getSpellListPageHref()`.
     - `getSpellModalId()` updated for `spell_id`.
   - Remove or stop using local `sortSpellsByName()` and `paginateSpells()` for API-backed pages because ordering and pagination now belong to the API contract.
   - Do not add a fallback to `src/data/spells/*.json`.

6. Add contract-focused tests.
   - Add `src/adapters/public-spell-api.test.ts` or equivalent Vitest coverage using mocked `fetch`.
   - Cover URL construction and response parsing for:
     - `GET /spell-types`
     - `GET /spells` with `page`, `pageSize`, `q`, `type`, `sourceName`, `sourceType`
     - `GET /spells/{spell_id}`
     - `GET /spell-suggestions` with `q`, `type`, and `limit`
   - Cover documented failures:
     - `400` invalid detail ID response preserves `invalid_request` and message.
     - `404` detail response preserves `not_found`.
     - `429` preserves `rate_limited` and `Retry-After`.
     - `503` preserves `service_unavailable` and `Retry-After`.
   - Cover malformed/non-JSON error response behavior as an adapter error.

7. Verify migrated paths.
   - Run `pnpm test` for adapter tests and existing test suite.
   - Run `pnpm build` to validate Astro SSR compilation and route code.
   - If the canonical public API is reachable from this environment, perform read-only smoke checks using `fetch`/browser-equivalent requests against:
     - `https://worldofaletheia.com/api/v1/spell-types`
     - `https://worldofaletheia.com/api/v1/spells?page=1&pageSize=1`
     - one valid `spell_id` from the list against `/api/v1/spells/{spell_id}`
     - an invalid detail ID to observe `400`
     - a UUIDv7-shaped missing ID to observe `404` if a safe known-missing ID is available.
   - Do not query D1 directly for spell verification from the main-site implementation.

## Assumptions And Gaps To Document During Implementation
- The canonical public API at `https://worldofaletheia.com/api/v1/*` is deployed and reachable before cutting over the main-site pages. If not reachable, implementation can still compile and adapter tests can pass, but runtime/build smoke verification is blocked.
- Current main-site pages do not have standalone spell detail URLs. The existing “detail” surface is a modal populated from list/type results. The adapter should still implement `getSpellById()` to satisfy the contract and future detail-page use, but no new public spell detail route is required unless requested.
- Current main-site filters are type pages only. The adapter should support all documented filters (`q`, `type`, `sourceName`, `sourceType`) so future UI changes stay localized, but this migration should not invent a new search/filter UI unless requested.
- Existing type URLs are slugified labels, while the API filters by exact type label. The migration preserves current URLs by resolving slugs from `listSpellTypes()` before calling `listSpells({ type })`.
- Existing local data used singular `spell_type` and string `keywords`; the API returns `spell_types: string[]` and `keywords: string[]`. Components will render joined labels while preserving the API field shape internally.
- Keeping `src/data/spells/*.json` files in the repo may still be acceptable as non-runtime artifacts, but migrated website reads must not import them.
- The public API contract does not expose per-spell `lastmod`; sitemap spell URLs will continue without spell-specific last modified dates.
