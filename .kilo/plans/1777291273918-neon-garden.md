# Plan: Ingest Expanded Sorcerer Spell Data

## Goal
Replace the current 676-spell sorcerer spell dataset with the new expanded dataset, regenerate the derived spell type list, and keep the existing `/systems/gurps/resources/sorcerer-spells` pages working with the larger data volume.

## Current Findings
- Current data lives in `src/data/spells/spells-raw.json` and has the shape `{ metadata, spells }`.
- Current derived type data lives in `src/data/spells/spell-types.json` as a sorted unique array of `spell_type` values.
- The consuming pages are:
  - `src/pages/systems/gurps/resources/sorcerer-spells/index.astro`
  - `src/pages/systems/gurps/resources/sorcerer-spells/all.astro`
  - `src/pages/systems/gurps/resources/sorcerer-spells/[type].astro`
- The route is `sorcerer-spells`, not `sorcer-spells`.
- The sample file `/tmp/sorcery-spells-flat-head.json` is intentionally truncated and not parseable as JSON, but the visible spell objects use the same field names as the current dataset:
  - `spell_name`
  - `spell_type`
  - `keywords`
  - `full_cost`
  - `casting_roll`
  - `range`
  - `duration`
  - `description`
  - `statistics`
- The sample metadata reports `totalSpells: 2207`, which is expected to supersede the current 676-spell file.
- The first sample spells already exist in the current dataset, supporting the assumption that the new file is a superset.
- The current site hardcodes the old `676` count in the spell source attribution section.
- Current modal IDs are derived from `spell_name` only. The existing 676-spell data already has duplicate spell names (`Eater`, `Strike Anosmic`), so the larger data may make duplicate modal IDs more likely.
- Existing `spell-types.json` matches `Array.from(new Set(spells.map(...))).sort()` using default JS sort.
- `.kilo/package-lock.json` exists alongside `pnpm-lock.yaml`; do not use `npm` or add dependencies.

## Implementation Plan

### 1. Add A Read-Only Validation/Generation Script
Create a small Node ESM script using only built-in modules, for example `scripts/generate-spell-data.mjs` or `scripts/generate-spell-types.mjs`.

The script should:
- Accept the source JSON path as an argument, defaulting to a chosen checked-in source path if provided.
- Parse the full new JSON file.
- Validate top-level shape: object with `metadata` and `spells` array.
- Validate every spell has required string fields used by the pages: `spell_name`, `spell_type`, `full_cost`, `casting_roll`, `range`, `duration`, `description`, `statistics`.
- Optionally treat `description` and `statistics` as displayable empty strings if the source has rare missing values, but report them in validation output.
- Normalize only where safe and visible in current data patterns:
  - Trim leading/trailing whitespace from string fields.
  - Preserve punctuation and typography inside spell text.
  - Do not rewrite prose, costs, or statistics.
- Generate `spell-types.json` from unique `spell_type` values sorted with default `.sort()` to preserve the existing type ordering behavior.
- Check for duplicate `spell_name` values and duplicate generated modal IDs, reporting them as warnings rather than blocking ingestion.
- Check for duplicate type slugs generated by the current slug algorithm and fail if collisions exist, because those would break `[type].astro` routing.
- Write regenerated files only when invoked during implementation, not during validation-only runs.

### 2. Replace Current Data With The New Superset
When the full source file is available:
- Use the full parseable new JSON file, not the truncated `/tmp/sorcery-spells-flat-head.json` sample.
- Replace `src/data/spells/spells-raw.json` with the validated generated output.
- Replace `src/data/spells/spell-types.json` with the regenerated type list.
- Preserve useful new metadata fields from the new source (`scrapedAt`, `totalSpells`, `sourceUrl`, `scrapingMethod`, `mergedAt`, `mergedFrom`, etc.) unless there is a reason to omit operational details.

### 3. Update The Sorcerer Spell Pages For Dynamic Counts
Update `src/pages/systems/gurps/resources/sorcerer-spells/index.astro` to import `spells-raw.json` and derive the displayed count from `spellsData.spells.length` or `spellsData.metadata.totalSpells` after validating they match.

This removes the stale hardcoded text:
- `These 676 sorcerer spells...`

Use the array length for the user-facing count, because it reflects what the site is actually rendering.

### 4. Harden Modal IDs And Sorting
Update modal ID generation in `all.astro` and `[type].astro` so each row/card gets a unique stable ID even when spell names repeat.

Recommended minimal approach:
- Sort spells into a new array without mutating imported JSON: `const spells = [...spellsData.spells].sort(...)`.
- Generate IDs using both index and slugified spell name in each rendered list, for example `spell-modal-${index}-${slug}`.
- Use the same computed ID for the button and dialog in the same `.map()` callback.

Why:
- Current duplicate spell names already produce duplicate modal IDs.
- The larger dataset likely increases this risk.
- This is a small local page fix, not a broader refactor.

### 5. Handle Minor Source Formatting Differences
The sample includes values like `None.` while the current pages suppress keywords only when `spell.keywords !== 'None'`.

Update keyword display checks in both spell pages to treat `None`, `None.`, and empty strings as absent. Keep the raw value in JSON unchanged.

Recommended helper local to each page or a tiny shared utility only if duplication becomes noisy:
- `const hasKeywords = (value: string | undefined) => Boolean(value?.trim() && !/^none\.?$/i.test(value.trim()));`

Given the logic appears in two pages only, keep it local unless implementation reveals more reuse.

### 6. Choose The Browsing/Data Delivery Strategy
The expanded dataset is about 2,207 spells. The current `all.astro` page renders one table row and one dialog per spell, which will likely produce a very large HTML response because every description and statistics block is duplicated into hidden dialog markup.

#### Option A: JSON At Build Time + One Large Static HTML Page
Keep importing `src/data/spells/spells-raw.json` directly and render all spells plus all modal detail content into `/systems/gurps/resources/sorcerer-spells/all`.

Pros:
- Minimal code change from current behavior.
- Fully static-rendered output and CDN-friendly delivery.
- No runtime database dependency or data sync path.
- Works offline after page load because every detail is already in the HTML.

Cons:
- Worst initial page weight: 2,207 rows plus 2,207 hidden dialogs with long prose/statistics.
- Browser must parse and retain a large DOM even if the user opens only one spell.
- Duplicate-name modal ID issues must be fixed before this is safe.
- Any later search/filter UI would still need to work around a very large DOM.

Assessment:
- Acceptable as a short-lived fallback, but not the best default for 2,207 spells.
- It optimizes implementation speed at the cost of user-perceived performance and HTML bloat.

#### Option B: JSON At Build Time + Static Index/Type/Paginated Pages
Keep the source as checked-in JSON, but make browsing primarily index-driven. Render type pages and/or paginated alphabetical pages, and avoid putting every full spell detail into one HTML page.

Possible shape:
- `/systems/gurps/resources/sorcerer-spells` remains the landing/type index.
- `/systems/gurps/resources/sorcerer-spells/all` becomes an alphabetical index with summary rows only, or redirects/links to page 1.
- Add static pagination such as `/systems/gurps/resources/sorcerer-spells/all/1`, `/all/2`, etc.
- Keep type pages static and sorted.
- Either render full detail only on the current page or add individual static detail pages if deep links are useful.

Pros:
- Preserves the existing JSON source of truth and static-first site model.
- Reduces per-page HTML and DOM size substantially.
- No Cloudflare D1 migration, seed, query, or operational drift risk.
- Type browsing already matches the mental model of this resource.
- Easy to validate with `pnpm build`.

Cons:
- Pagination improves load size but does not by itself solve search.
- Pure static filtering by query parameters is awkward unless client-side JS is added.
- If users frequently need arbitrary name lookup, an index-only UX may still feel slow without search.

Assessment:
- Best near-term fit. It solves the immediate 2,207-spell scale problem without adding runtime infrastructure.
- Prefer this if the goal is reliable ingestion and usable browsing now.

#### Option C: JSON At Build Time + Client-Side Filtering/Pagination
Build or publish a browser-consumable JSON artifact and use vanilla TypeScript to filter, paginate, and show one reusable detail panel/dialog on the client.

Pros:
- Avoids a huge static HTML document while enabling fast in-page filtering.
- No D1 migration or server query path.
- One reusable dialog avoids duplicate modal IDs and hidden DOM bloat.
- Can be progressively enhanced from a static index.

Cons:
- Ships the full spell dataset to the browser, which may still be several MB before compression.
- Adds client JavaScript to a page that can otherwise remain static.
- Search/filter behavior must be implemented and tested for accessibility and no-JS fallback.
- Canon and Using Aletheia remain no-island/static-first by policy; this GURPS resource is static content, so keep any script vanilla and small if used.

Assessment:
- Good if quick search/filter is a product requirement for this page.
- Not necessary for the initial ingestion unless static pagination/type pages are insufficient.

#### Option D: New D1 Spell Table
Create a dedicated D1 table for spells, ingest the new source into D1, and query it at runtime for pagination/filter/search.

Pros:
- Best fit for server-side pagination, filtering, faceting, and eventual full-text search.
- Avoids shipping the entire dataset or rendering all details in one HTML page.
- Aligns directionally with ADR-0011, which chooses D1 metadata indexes for scalable discovery and search.
- The app already has Cloudflare/D1 plumbing and repository examples under `src/lib`.

Cons:
- Highest implementation and operational complexity for this resource.
- Requires schema migration, ingestion/seed script, local/staging/prod D1 workflows, and drift checks.
- Adds a runtime dependency for a public spell reference that can currently be served from static JSON.
- Requires deciding whether JSON or D1 is canonical. Keeping both as active sources creates drift risk.
- Requires API/page query code and tests; this is no longer just a data-file ingestion.

Assessment:
- Justified if spells become an interactive/search-heavy resource with server-side search, user-specific visibility, saved selections, or frequent independent updates.
- Overkill for the immediate replacement unless there is a firm requirement for runtime search/filtering now.

#### Recommendation
Use Option B for this ingestion: keep checked-in JSON as the source, regenerate `spell-types.json`, and change browsing to avoid rendering every full spell into one `all` page.

Recommended implementation target:
- Keep direct JSON imports for build-time rendering.
- Make the landing page and type pages the primary navigation.
- Change the all-spells route to a lightweight alphabetical index or statically paginated pages.
- Render full spell details only for the currently visible page or on per-spell static detail pages.
- Do not introduce D1 for spells in this iteration.

Defer Option C or D until a concrete requirement appears:
- Choose Option C if users need instant in-page search/filter over the public spell list and the full JSON payload is acceptable.
- Choose Option D if spell search/filter must be server-side, D1-backed, operationally indexed, or eventually tied to authenticated campaign features.

### 7. Adjust Pages For The Chosen Static Strategy
If using the recommended Option B:
- Update `all.astro` so it does not render all spell detail dialogs into a single page.
- Add static pagination or split detail pages before replacing the dataset.
- Keep type pages as static filtered pages, but ensure they only render detail dialogs for the spells on that type page.
- Consider individual static spell detail pages only if stable deep links are useful enough to justify slug collision handling.

Run these checks after implementation:

- Confirm at least one regenerated type page works, such as `/systems/gurps/resources/sorcerer-spells/adventurer-spells`.
- Confirm `None.` keywords do not display as meaningful keywords.

## Open Assumptions
- The full new file will have the same top-level shape as the sample: `{ metadata, spells }`.
- The new file is intended to fully replace, not merge with, the old file.
- No new dependencies should be added.
- Attribution to Generic Universal Eggplant remains appropriate.

## Implementation Notes
- Use `pnpm`, never `npm` or `yarn`.
- Use path aliases in Astro imports.
- Keep changes limited to spell data, the spell pages, and the small generation/validation script.
- Do not populate `src/services`, `src/adapters`, or `src/contracts` for this; direct JSON imports are sufficient.
